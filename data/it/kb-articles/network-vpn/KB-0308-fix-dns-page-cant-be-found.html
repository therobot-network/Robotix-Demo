Fix DNS and "Page Can't Be Found" Errors

Summary

"Page can't be reached," "ERR_NAME_NOT_RESOLVED," "DNS_PROBE_FINISHED_NXDOMAIN," or "Safari Can't Find the Server" errors usually mean your device cannot translate a website name (like intranet.robotix.com) into an IP address. This is a DNS (Domain Name System) issue. The most common causes are a stale DNS cache, incorrect DNS server settings, GlobalProtect VPN not being connected (for internal addresses), or a temporary ISP or office DNS outage. This article resolves the most common scenarios.

Before You Begin

  • Confirm you have a working network connection. Try loading https://1.1.1.1 in a browser. If that loads but named sites do not, the problem is definitely DNS.
  • Note whether the broken site is a public site (e.g., google.com) or an internal Robotix IT site (e.g., intranet.robotix.com). The fixes differ slightly.
  • For internal Robotix IT sites: confirm GlobalProtect VPN is connected if you are off the corporate network. DNS for internal names is only resolved through the VPN tunnel.

Steps

Step 1 — Reload the page and clear the browser cache

Start with the simplest fixes:

  1. Press F5 (Windows) or Cmd + R (Mac) to reload the page.
  2. If that does not work, do a hard reload: Ctrl + Shift + R (Windows) or Cmd + Shift + R (Mac).
  3. If still broken, clear the browser's DNS cache: - Chrome / Edge: Open a new tab, go to chrome://net-internals/#dns, click Clear host cache, then go to chrome://net-internals/#sockets and click Flush socket pools. - Safari (macOS): Go to Develop menu > Empty Caches (enable the Develop menu in Safari Settings > Advanced if it is not visible).
  4. Close and reopen the browser. Retry the page.

Step 2 — Flush the operating system DNS cache

The OS maintains its own DNS cache separate from the browser.

Windows 11: 1. Right-click the Start button and select Terminal (Admin) or Command Prompt (Admin). 2. Run the following commands one at a time: ipconfig /flushdns ipconfig /registerdns 3. You should see: "Successfully flushed the DNS Resolver Cache." 4. Retry the page.

macOS (Sonoma / Sequoia): 1. Open Terminal (Spotlight: Cmd + Space > type "Terminal"). 2. Run: sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder 3. Enter your Mac's local password and press Return. 4. Retry the page.

Step 3 — For internal Robotix IT sites — check GlobalProtect VPN

Robotix IT's internal DNS servers (which resolve *.robotix.com and other private hostnames) are only accessible when you are: - Connected to the office network (Robotix-Corp Wi-Fi or wired Ethernet), or - Connected via GlobalProtect VPN.

  1. Check the GlobalProtect icon in the system tray (Windows) or menu bar (Mac).
  2. If it is grey or shows "Disconnected," click it and select Connect.
  3. Approve the Okta Verify MFA push.
  4. Once GlobalProtect shows the green "Connected" state, retry the internal site.
  5. If GlobalProtect won't connect, see KB-0304.

Step 4 — Verify DNS server addresses (Windows)

Your device should be receiving DNS server addresses automatically from DHCP. If they have been changed manually, DNS may break.

  1. Open Settings > Network & internet > Wi-Fi (or Ethernet) > click your active connection > Hardware properties.
  2. Under DNS server assignment, verify it shows Automatic (DHCP). If it shows manual IPs that are not Robotix IT's DNS servers (10.1.1.53, 10.1.2.53), click Edit.
  3. Set DNS settings to Automatic (DHCP) and save.
  4. Flush the DNS cache again (Step 2) and retry.

Step 5 — Verify DNS server addresses (macOS)

  1. Open System Settings > Network.
  2. Click your active connection (Wi-Fi or Ethernet) > Details.
  3. Click the DNS tab.
  4. If you see any manually entered DNS servers that do not belong to Robotix IT (10.1.1.53, 10.1.2.53, or 8.8.8.8 as a fallback), remove them (click the minus button) and save.
  5. If using DHCP, the DNS servers should be assigned automatically. Click Renew DHCP Lease on the TCP/IP tab.
  6. Retry after flushing the cache (Step 2).

Step 6 — Test DNS resolution manually

You can test whether your device can resolve a name and which DNS server it is querying.

Windows 11:

nslookup google.com
nslookup intranet.robotix.com

Look at the Server: line — it should show a Robotix IT DNS server (10.1.1.53 or 10.1.2.53) when on VPN or the office network, or your ISP/router's DNS when off-network for public names.

macOS:

nslookup google.com
nslookup intranet.robotix.com

Or using dig:

dig google.com
dig intranet.robotix.com

If nslookup returns NXDOMAIN or can't find, the DNS record is not resolving — move to Step 7.

Step 7 — Restart your network adapter

Windows 11: 1. Go to Settings > Network & internet > Advanced network settings. 2. Click your active adapter (e.g., Wi-Fi) > Disable. 3. Wait 10 seconds, then click Enable.

macOS: 1. Hold Option and click the Wi-Fi menu bar icon. 2. Click Turn Wi-Fi Off, wait 10 seconds, then Turn Wi-Fi On. 3. Reconnect to Robotix-Corp.

Step 8 — Try a known-good DNS server temporarily (public sites only)

If you are troubleshooting public internet access only (not internal sites) and nothing else works, you can temporarily configure a public DNS server to test.

Note: Do not use public DNS servers (e.g., 8.8.8.8) as a permanent fix for internal Robotix IT name resolution — this will not work for *.robotix.com addresses and may create security exceptions. Use this only to confirm whether the issue is your ISP's DNS.

Windows: Settings > Network > your adapter > DNS assignment > Manual > IPv4 Primary: 8.8.8.8, Secondary: 1.1.1.1. Test, then revert to Automatic (DHCP).

macOS: System Settings > Network > adapter > Details > DNS > add 8.8.8.8. Test, then remove and revert.

If public sites now resolve with an alternate DNS, your router or ISP's DNS is the issue — restart your home router or contact your ISP.

Common DNS Error Messages

Browser Error Meaning
ERR_NAME_NOT_RESOLVED (Chrome/Edge) DNS query returned no result
DNS_PROBE_FINISHED_NXDOMAIN (Chrome) Hostname does not exist in DNS
DNS_PROBE_FINISHED_NO_INTERNET (Chrome) No DNS server reachable
"Safari Can't Find the Server" macOS DNS resolution failed
"Server Not Found" (Firefox) DNS failure or server offline
SERVFAIL in nslookup DNS server returned an error (server issue)
